# Mesh — Architecture Spec

## Project Overview

**Mesh** is a distributed, Cypher-compliant graph database written in Rust. Supports horizontal scaling via configurable clusters with graph data partitioned across multiple nodes.

## Workspace Layout

```
mesh/
├── Cargo.toml              # Workspace root
├── crates/
│   ├── meshdb-core/          # Node, Edge, Property types, ID generation
│   ├── meshdb-storage/       # RocksDB-backed store, indexes, transactions
│   ├── meshdb-cypher/        # Lexer, parser, logical plan, optimizer
│   ├── meshdb-executor/      # Physical operators, iterator model
│   ├── meshdb-cluster/       # Raft, partitioning, membership, replication
│   ├── meshdb-rpc/           # tonic/gRPC service definitions + client
│   ├── meshdb-bolt/          # Bolt v4.4–5.4 packstream, messages, framing
│   ├── meshdb-client/        # In-process + Bolt client library
│   ├── meshdb-tck/           # openCypher TCK harness (cucumber-driven)
│   └── meshdb-server/        # Binary: config, startup, Bolt + gRPC listeners
```

## Architecture Decisions

### Storage Engine (`meshdb-storage`)
- **RocksDB** via `rust-rocksdb` crate as the backing store (same approach as Dgraph, Nebula Graph, ArangoDB).
- Column families for: nodes, edges (forward adjacency), edges (reverse adjacency), properties, indexes.
- Each node/edge gets a globally unique 128-bit ID.
- Edges stored twice (forward + reverse) for fast bidirectional traversal.
- Transactions use RocksDB WriteBatch + snapshot isolation (single-partition initially).

### Core Data Model (`meshdb-core`)
- `NodeId` / `EdgeId` — 128-bit unique identifiers (UUID v7 or similar for ordering).
- `Property` — enum covering: `Null`, `String`, `Int64`, `Float64`, `Bool`, `List`, `Map`, temporal (`DateTime`, `LocalDateTime`, `Date`, `Time`, `Duration`), and spatial (`Point` — Cartesian 2D/3D, WGS-84 2D/3D, EPSG-tagged).
- `Node` — id, labels (`Vec<String>`), properties (`HashMap<String, Property>`).
- `Edge` — id, edge_type (String), source NodeId, target NodeId, properties.

### Cypher Query Engine (`meshdb-cypher`)
- Hand-rolled recursive-descent parser emitting a logical plan (tree of relational + graph operators). Planner converts logical → physical plan.
- Key logical operators: NodeScan, EdgeExpand, Filter, Project, Aggregate, Sort, Limit.
- **Implemented Cypher surface:**
  - Read: `MATCH`, `OPTIONAL MATCH`, `WHERE`, `RETURN` (with aliases, `DISTINCT`), `WITH` (re-projection + filter), `ORDER BY`, `LIMIT`, `SKIP`, `UNION` / `UNION ALL`.
  - Write: `CREATE`, `MERGE` (with `ON CREATE SET` / `ON MATCH SET`), `SET` (property, `+=` merge, label), `REMOVE` (property, label), `DELETE` / `DETACH DELETE`, `FOREACH`.
  - Flow: `UNWIND`, `CASE ... WHEN ... ELSE ... END` (simple and generic), parameters (`$name`).
  - Patterns: variable-length paths `()-[*1..3]->()`, Neo4j 5 quantifier shorthand (`->+` = `*1..`, `->*` = `*0..`, `->{n}` / `->{n,m}` / `->{n,}` / `->{,m}`), `shortestPath(...)` and `allShortestPaths(...)` — both share a single layered-BFS operator that builds a parent DAG and enumerates one or all minimum-length paths.
  - Expressions: list comprehensions, pattern comprehensions, `reduce`, quantifier predicates (`all`/`any`/`none`/`single`), `EXISTS { ... }`, `COUNT { ... }`, and `COLLECT { ... }` subquery expressions.
  - Procedures / subqueries: `CALL { ... }` (unit and returning), `CALL proc YIELD ...` against a runtime-extensible registry in `meshdb-executor` with built-in `db.labels()` / `db.relationshipTypes()` / `db.propertyKeys()` / `db.constraints()` installed by default, plus a growing set of opt-in APOC procedures behind `apoc-*` features (see *APOC compatibility* below). Procedure name segments can coincide with Cypher reserved words (`apoc.create.node`, `apoc.merge.*` etc.).
  - Schema: `CREATE INDEX` / `DROP INDEX` / `SHOW INDEXES` on node label+property tuples (`FOR (n:Label) ON (n.prop)` or composite `ON (n.a, n.b, ...)`) *and* relationship type+property tuples (`FOR ()-[r:TYPE]-() ON (r.prop)` or composite); the planner uses composite indexes for prefix-matched equality seeks (`MATCH (n:L {a: 1, b: 2})` or WHERE conjuncts). Edge property predicates with both endpoints unbound lower to `EdgeSeek` through the edge index. `CREATE POINT INDEX` / `DROP POINT INDEX` / `SHOW POINT INDEXES` on single-property `Property::Point` columns for both node and relationship scope — backed by a Z-order (Morton) cell quantizer in separate CFs with per-SRID domains; the planner rewrites `WHERE point.withinbbox(var.p, lo, hi)` and `WHERE point.distance(var.p, center) <[=] r` into `PointIndexSeek` (node scope) or `EdgePointIndexSeek` (relationship scope, on unbound-endpoints patterns `MATCH ()-[r:T]-()`), with the distance predicate kept as a residual Filter to cull the enclosing-bbox overshoot. `CREATE CONSTRAINT` / `DROP CONSTRAINT` / `SHOW CONSTRAINTS` for UNIQUE, NOT NULL, `IS :: <TYPE>` (STRING/INTEGER/FLOAT/BOOLEAN), and composite `IS NODE KEY` — node scope `FOR (n:Label)` or relationship scope `FOR ()-[r:TYPE]-()` — with optional name + `IF [NOT] EXISTS`, replicated across Raft and routing clusters; UNIQUE and NODE KEY both auto-provision a backing property index (composite for NODE KEY) so enforcement stays O(log N) per insert; built-in `db.constraints()` procedure.
  - Scalars: full openCypher scalar surface (string, math, temporal, spatial) plus the widely-expected Neo4j extensions (`*OrNull`, `*List`, `valueType`, `randomUUID`, `round` with precision+mode, `char_length`).

### APOC compatibility (`meshdb-apoc` + executor procedures)

APOC-compatible scalars live in the standalone `meshdb-apoc` crate; write / introspection procedures live beside the `db.*` built-ins in `meshdb-executor::procedures`. The full APOC surface is **enabled by default** — `meshdb-server`, `meshdb-executor`, and `meshdb-apoc` all include the umbrella `apoc` (or `full`) feature in their `default = [...]` lists. Embedded callers who want a slimmer build can disable defaults and re-enable specific namespaces (`apoc-coll`, `apoc-text`, etc.). Shipped today:

- **Scalars** — `apoc.coll.*` (sum / avg / max / min / toSet / sort[Desc] / reverse / contains / union / intersection / subtract / flatten / zip / indexOf / occurrences / toMap), `apoc.text.*` (join / split / replace / indexOf / lpad / rpad / capitalize[All] / decapitalize / swapCase / camelCase / snakeCase / upperCamelCase / repeat / reverse / urlencode / urldecode / regexGroups / hexValue / base64Encode / base64Decode / byteCount / clean / levenshteinDistance), `apoc.map.*` (merge / fromPairs / fromLists / fromValues / setKey / removeKey / removeKeys / values / submap / mergeList), `apoc.util.*` (md5 / sha1 / sha256 / sha384 / sha512), `apoc.convert.*` (toJson / fromJsonMap / fromJsonList), `apoc.date.*` (currentTimestamp / toISO8601 / fromISO8601 / convert / add), `apoc.number.*` (parseInt / parseFloat / arabicToRoman / romanToArabic / format), `apoc.create.*` scalars (uuid / uuidBase64 / uuidBase64ToHex / uuidHexToBase64), `apoc.meta.*` scalars (type / isType / types).
- **Aggregates** (always-on, wired into the native aggregate operator): `apoc.agg.first` / `last` / `nth` / `median` / `product`.
- **Write procedures** — behind `apoc-create`: `apoc.create.node(labels, props)`, `apoc.create.relationship(from, type, props, to)`, `apoc.create.addLabels` / `removeLabels` / `setLabels` / `setProperty` / `setRelProperty`. Behind `apoc-refactor`: `apoc.refactor.setType(rel, newType)`. All route through the procedure registry's write-dispatch path, which passes the `GraphWriter` alongside args — so writes ride the same single-node / Raft / routing-2PC machinery as `CREATE` / `MERGE`.
- **Read procedures** — behind `apoc-meta`: `apoc.meta.schema()`.
- **Batched-write procedures** — `apoc.periodic.iterate(iterateQuery, actionQuery, config)` ships in-tree (no Cargo feature) as a planner-level rewrite into a dedicated `LogicalPlan::ApocPeriodicIterate` node that the server-side dispatcher unfolds into per-batch commits. Honors `batchSize`, `params`, `retries`, `failedParams`, `iterateList`, `parallel`, and `concurrency` (default 50 when `parallel: true`); emits the standard 13-column Neo4j result row (`batches`, `total`, `committedOperations`, `failedOperations`, `failedBatches`, `errorMessages`, `failedParams`, `batch`, `operations`, `updateStatistics`, `retries`, `timeTaken`, `wasTerminated`). Same explicit-tx rejection as `CALL { } IN TRANSACTIONS` — both forms commit per batch and conflict with an enclosing `BEGIN`/`COMMIT`. `parallel: true` fans batches out through a `tokio::sync::Semaphore` worker pool with `Arc<Mutex<...>>`-guarded stats accumulators; throughput gains materialize fully in single-node mode, while Raft and routing-2PC modes still serialize commits internally so the win there is limited to hiding per-batch planning / read latency.
- **Path procedures** — behind `apoc-path`: `apoc.path.expand`, `apoc.path.expandConfig`, `apoc.path.subgraphAll`, `apoc.path.subgraphNodes`, `apoc.path.spanningTree`. Share a streaming BFS walker that honours the full APOC filter DSL (relationshipFilter `>TYPE|<TYPE`, labelFilter `+include|-exclude|>end|/terminator`) and all seven uniqueness modes. `apoc-path` also enables the Path-shaping scalars (`create` / `slice` / `combine` / `elements`).
- **Cypher-runner procedures** — behind `apoc-cypher`: `apoc.cypher.run(cypher, params)` (read-only — rejects plans containing write operators) and `apoc.cypher.doIt(cypher, params)` (allows writes, which accumulate in the outer query's buffered writer for atomic commit). Inner executions thread the same reader / writer / procedure registry so nested `CALL` sites resolve naturally.
- **File I/O procedures** — behind `apoc-load`: `apoc.load.json(urlOrPath, path)`, `apoc.load.csv(urlOrPath, config)`. Behind `apoc-export`: `apoc.export.{csv,json,cypher}.{all,query}`. Both directions share a strict-default-off `ImportConfig` security surface (`enabled` / `allow_file` / `allow_http` / `file_root` / `url_allowlist` / `allow_unrestricted`) set via the `[apoc_import]` section of the server config. The single `resolve_source` / `resolve_export_path` choke points enforce every gate before any I/O handle is opened.
- **Trigger procedures** — behind `apoc-trigger`: install / drop / list / start / stop, all five APOC-compatible. Registered triggers fire on commit boundaries with the tx diff exposed via the full Neo4j APOC param surface: `$createdNodes` / `$createdRelationships` / `$deletedNodes` / `$deletedRelationships` (lists) plus `$assignedNodeProperties` / `$removedNodeProperties` / `$assignedRelationshipProperties` / `$removedRelationshipProperties` / `$assignedLabels` / `$removedLabels` (lists of `{key, old, new, node|relationship}` and `{label, node}` records). All four phases work — `before` (fires pre-commit, errors abort the tx, writes commit atomically with the user's writes), `after` (fires post-commit, sync), `afterAsync` (fires post-commit in a tokio::spawn so it doesn't block the response), and `rollback` (fires when the commit fails, including before-trigger aborts). Triggers persist in a dedicated `trigger_meta` RocksDB CF and survive restart. Install/drop/start/stop emit `GraphCommand::InstallTrigger` / `DropTrigger` through the writer so the cluster commit path replicates them across every peer: Raft mode rides the existing log + per-peer applier (with a registry-cache refresh hook on `StoreGraphApplier`), routing mode rides the DDL fan-out via two `InstallTrigger` / `DropTrigger` gRPC methods on `MeshWrite`. Firing is leader/coordinator-only — exactly one peer per logical commit fires. Trigger-induced writes recurse through `commit_buffered_commands` with a `from_trigger = true` flag that suppresses further firing, so cluster-mode trigger writes replicate through the same Raft / 2PC machinery the originator's writes did. A thread-local suppression guard catches the sync-path recursion case as a backstop. The `databaseName` arg is accepted but ignored since Mesh is single-db today.
- **Not yet implemented:** none of the remaining APOC procedures are on the near-term roadmap; each represents a separate infrastructure investment rather than more of the same extension pattern.

### Query Execution (`meshdb-executor`)
- Volcano/iterator (pull-based) model — each operator implements `next() -> Option<Row>`.
- Operators: Scan, IndexSeek (composite-capable — tuple prefix seeks on the index CF), EdgeSeek (relationship-scope analogue for unbound-endpoints patterns), PointIndexSeek + EdgePointIndexSeek (Z-order bbox range + SRID-aware enclosing-bbox for distance queries, node and relationship scope), Expand, ShortestPath (layered BFS with multi-parent DAG; emits one path for `shortestPath` or every minimum-length path for `allShortestPaths`), Filter, Project, Aggregate, Sort, Limit, CreateNode, CreateEdge, Delete.
- Later optimization: vectorized execution for analytical queries.

### Distribution / Clustering (`meshdb-cluster`)
- **Partitioning:** FNV-1a hash over NodeId maps each node to a `PartitionId`. Edges live on the partition that owns the source vertex, with ghost/stub copies on the target's partition for reverse traversal.
- **Consensus:** Raft via `openraft` 0.9 for metadata and write coordination — full `RaftStorage` impl, AppendEntries / Vote / InstallSnapshot wired up.
- **Reads:** Any node can serve a query; multi-partition traversals scatter-gather through `meshdb-rpc::partitioned_reader`.
- **Writes (routing mode):** Multi-peer Cypher writes ride a durable 2PC. Coordinator side logs PREPARE / CommitDecision / AbortDecision / Completed to `coordinator-log.jsonl`; participant side logs PREPARE / Committed / Aborted to `participant-log.jsonl` and fsyncs before each RPC ACKs. Both logs survive independent coordinator/participant crashes — on startup each peer rehydrates staging from its log, polls peers via `ResolveTransaction` to learn the coordinator's decision for any in-doubt txid, and only then opens the gRPC listener. Per-phase RPC timeouts (10s PREPARE / 30s COMMIT / 10s ABORT by default) bound a stalled peer's ability to block the round. PREPARE retries with identical commands are idempotent; conflicting payloads fail loudly.
- **Writes (multi-raft mode):** `mode = "multi-raft"` opt-in. Each partition has its own openraft group, replicated across `replication_factor` peers (default `min(3, peers.len())`); DDL + cluster membership ride a separate metadata Raft group spanning every peer. Single-partition writes arriving on a non-leader peer are server-side proxied to the partition leader via `MeshWrite::ForwardWrite`; DDL through `MeshWrite::ForwardDdl` (no client-visible redirects, transparent to Bolt sessions). Cross-partition writes use a Spanner-style 2PC where both PREPARE and COMMIT are proposed through the partition Raft (`GraphCommand::PreparedTx` / `CommitTx` / `AbortTx`) — staged state is replicated by the time PREPARE-ACK returns, so a partition leader crash post-PREPARE can't lose staging. The coordinator log records the cross-partition decision; new partition leaders use `ResolveTransaction` polling against every peer's log to resolve in-doubt PREPAREs after a coordinator crash. A periodic recovery loop (`DEFAULT_RECOVERY_INTERVAL`, 60s) re-runs in-doubt resolution post-startup so a coordinator that crashes mid-flight while the cluster stays up still converges. **DDL barrier:** every peer tracks `min_meta_index`, advanced on local DDL propose and on inbound forward_write/forward_ddl; partition writes await the local meta replica catching up before proposing, so a `CREATE INDEX` issued through any peer is guaranteed visible by the time a follow-up write applies. **Per-partition snapshots:** `PartitionGraphApplier::snapshot` packs only that partition's nodes + edges + pending-tx rows (~1/N of cluster data) instead of the full-graph rocksdb checkpoint; restore validates partition_id and rebuilds in-memory pending_txs from the freshly-restored on-disk state. Wire-protocol gets a typed `oneof target { meta, partition }` in `RaftRpcRequest` so a single `MeshRaftService` dispatches to the right group via `RaftGroupRegistry`. Group-aware on-disk layout: `data_dir/raft/meta/` plus `data_dir/raft/p-<N>/` per replicated partition. **Dynamic rebalancing:** `MultiRaftCluster::add_partition_replica` / `remove_partition_replica` wrap openraft's `change_membership` for per-partition voter changes; on success, a `ClusterCommand::SetPartitionReplicas` rides the meta Raft so every peer's persisted view of placement updates and a restart picks up the new replica set. Runtime partition group spin-up is supported — `meshdb_server::instantiate_partition_group` creates the rocksdb dir, builds the `RaftCluster` + `PartitionGraphApplier`, and registers the new group with both `MultiRaftCluster` (`Arc<RwLock<HashMap>>`-backed for runtime mutability) and the dispatch `RaftGroupRegistry` (also `RwLock`-backed) without restart. Multi-raft benchmarks live at `crates/meshdb-server/benches/multi_raft_bench.rs` covering single-partition local-leader, single-partition forwarded, and cross-partition 2PC commit paths.
- **Partition count change (resharding):** `num_partitions` is fixed at boot. Hash partitioning means every key hashes to exactly one of `num_partitions` slots, so changing the count silently invalidates every key's partition assignment — `meshdb_server::check_num_partitions_marker` writes a `data_dir/num_partitions` marker on first start and refuses to bring up the cluster if a subsequent restart names a different value. To reshard, the operator dumps the graph (`apoc.export.cypher.all` is the typical path), wipes every peer's `data_dir`, and re-imports against a fresh cluster started with the new `num_partitions`. Online resharding (live key migration without dump-and-restore) is its own multi-week project that requires dual-routing during the transition + Raft-group lifecycle for new partitions; intentionally out of scope for v2.
- **Leader balancing (manual):** the cluster reports
  `mesh_multiraft_partitions_led` per peer so operators can detect
  imbalance via Prometheus. Rebalancing today is operator-driven:
  call `MeshService::drain_leadership` on the over-loaded peer (or
  restart it). True automatic rebalancing requires a real
  leader-transfer primitive (TimeoutNow / `transfer_leader`) which
  openraft 0.9 doesn't expose — `force_election` races the source's
  heartbeats and can leave the cluster oscillating. Future work:
  vendor a fork or wait for upstream to land the primitive, then
  build a periodic balancer on top.
- **Reads (multi-raft mode):** `read_consistency = "linearizable"` opts both the point-read path (`get_node`, `outgoing`, `incoming`) and the scatter-gather path (`nodes_by_label`, `nodes_by_property`, `edges_by_property`, `all_node_ids`, `get_edge`) into a partition-leader-fenced fan-out. Point reads forward to the partition leader via `MultiRaftCluster::leader_of`; scatter reads dispatch to every peer with `linearizable=true`, each receiver fences on the partitions it leads via `Raft::ensure_linearizable` and filters its local-store reply to those partitions, so the union covers every partition exactly once with no duplicates and no stale follower data. Default (`local`) reads from the closest replica's local store.
- **Bolt routing-table refresh:** `routing_ttl_seconds` controls the TTL advertised in every Bolt ROUTE response. Multi-raft mode defaults to 30s so drivers re-fetch as partition leadership shuffles or membership changes; other modes keep the historical effectively-infinite TTL.
- **OpenTelemetry / OTLP tracing:** `[tracing] otlp_endpoint = "..."` plumbs every `tracing::span` from the server (gRPC handlers, Raft applies, the Cypher executor) through the OpenTelemetry SDK's batch span processor onto an OTLP/gRPC collector. `service_name` (default `meshdb-server`) and `sample_rate` (default 1.0) are operator-tunable.
- **Cluster-wide consistent backup:** `MeshService::take_cluster_backup` takes a snapshot on every Raft group across every peer in parallel via a new `MeshWrite::TakeBackup` RPC. Each peer's snapshot lives in its own `data_dir/raft/<group>/` rocksdb under stable keys; the orchestrator returns a manifest of (group, peer) entries plus errors. Cross-group consistency at restore is provided by the existing 2PC recovery loop — any in-doubt PREPAREs in the per-partition snapshots converge via `ResolveTransaction` polling at startup. Restore is operator-driven: stop the cluster, replace each peer's `data_dir`, restart.
- **TLS cert hot-reload:** `[bolt_tls] reload_interval_seconds = N` and `[grpc_tls] reload_interval_seconds = N` enable a background reload task that polls cert + key mtimes and hot-swaps the in-memory `CertifiedKey` via a `ResolvesServerCert` impl when either file changes. New TLS handshakes pick up the rotated cert immediately; in-flight handshakes keep the cert they completed against. Reload errors log at WARN and the previous cert stays in place. The gRPC path bypasses tonic's `ServerTlsConfig` (which doesn't expose the rustls cert resolver) and terminates TLS in a custom `tokio_rustls::TlsAcceptor` in front of `serve_with_incoming_shutdown`. Both paths share `meshdb_server::tls_reload`.
- **Future:** LDG streaming partitioner as an alternative to hash partitioning. Hot data migration end-to-end E2E test (the API surface composes; an integration test that demonstrates openraft's InstallSnapshot catching up a freshly-spun-up replica needs more harness work — sharing storage between the running peer and the new partition Raft). Online resharding. Auto leader balancer.

### Inter-Node Communication (`meshdb-rpc`)
- **gRPC** via `tonic` crate.
- Service definitions:
  - `WriteService` — forward writes to partition owner
  - `QueryService` — scatter-gather for cross-partition reads
  - `ClusterService` — membership, health checks, leader election
  - `SnapshotService` — snapshot transfer for new/recovering nodes
- Protobuf definitions in a shared `proto/` directory.

### Server (`meshdb-server`)
- Binary entry point.
- Config via TOML file (listen address, cluster peers, data directory, RocksDB tuning), with common knobs also settable via CLI flags / env vars.
- Bolt protocol listener (`meshdb-bolt`) supports handshake v4.4 through 5.4 — HELLO, RUN, PULL, BEGIN/COMMIT/ROLLBACK, RESET, GOODBYE, plus ROUTE for cluster-aware drivers. TLS is available for both Bolt and gRPC listeners via `tokio_rustls`.
- Neo4j drivers can connect directly; the `meshdb-client` crate exposes the same surface in-process for embedded use.

## Implementation Status

The foundational phases are all shipping — single-node graph store, Cypher parser + executor, gRPC cluster RPCs, Raft-backed cluster membership, and a Bolt-speaking server. Multi-raft (`mode = "multi-raft"`) ships as an opt-in fourth cluster mode alongside single / routing / raft: per-partition openraft groups + a metadata Raft group for DDL, server-side `MeshWrite::ForwardWrite` proxying so Bolt clients see one endpoint regardless of partition leadership, Spanner-style PREPARE-Raft for cross-partition writes (`GraphCommand::PreparedTx`/`CommitTx`/`AbortTx` ride the partition Raft, no in-doubt window dependent on a participant log fsync), and `recover_multi_raft_in_doubt` driving in-doubt PREPAREs through `ResolveTransaction` polling on startup. Tier-2 production-readiness items are also in: linearizable reads (`read_consistency = "linearizable"`, point + scatter end-to-end), Bolt routing-table refresh on a 30-second TTL by default in multi-raft, OpenTelemetry/OTLP tracing via `[tracing]`, cluster-wide consistent backups via `MeshService::take_cluster_backup` with serializable JSON manifests + a `meshdb-server validate-backup` subcommand, and TLS cert hot-reload (Bolt + gRPC). Tier-A operational hardening: graceful drain on SIGTERM / SIGINT (every partition this peer leads steps down before the listener exits, bounded by `shutdown_drain_timeout_seconds`), per-query timeout (`query_timeout_seconds` wraps every Cypher RUN in `tokio::time::timeout` so runaway queries surface `DeadlineExceeded` instead of hanging), and gRPC mTLS (set `[grpc_tls] client_ca_path` and the listener requires every inbound TLS handshake to present a client cert chained to the bundle; outbound peer endpoints automatically present the configured identity). Tier-B operational hardening: k8s `/livez` and `/readyz` HTTP endpoints (the shutdown path flips readiness *before* draining so the kubelet stops sending traffic in time), result-set max-rows cap (`query_max_rows` returns `ResourceExhausted` instead of OOMing on unbounded MATCH), per-peer Cypher concurrency cap (`max_concurrent_queries`), and durable JSONL audit log (`audit_log_path` records every `drain_leadership` / `take_cluster_backup` call with timestamp + initiator + structured args + success/error, fsync'd before return). Tier-C performance + ergonomics: Cypher plan cache (`plan_cache_size`) keyed on the schema fingerprint so repeated parametrised queries skip parse + plan and stale plans self-correct on every peer when DDL changes the catalog, RocksDB tuning knobs (`[storage]` exposes `max_open_files`, `write_buffer_size_bytes`, `max_write_buffer_number`, `bloom_filter_bits_per_key`, `keep_log_file_num`), plus production-shaped deploy artifacts under `deploy/`: k8s StatefulSet + Service + PDB + ConfigMap with anti-affinity + livez/readyz probes, an importable Grafana dashboard, Prometheus alert rules, and an operator runbook covering rolling upgrades / drain / backup-restore / TLS rotation. 2PC across partitions is hardened end-to-end: durable coordinator + participant logs, per-phase timeouts, idempotent PREPARE retries, and cross-peer decision recovery via `ResolveTransaction` so coordinator and participant crashes recover independently without operator intervention — the four recovery branches (coordinator crash pre-decision, coordinator crash mid-COMMIT fanout, participant crash after PREPARE-ACK + `ResolveTransaction` recovery, and synchronous PREPARE-rejection rollback) are all exercised by deterministic fault-injection integration tests in `crates/meshdb-server/tests/two_pc_fault_injection.rs`. The planner-side index work is caught up: composite property indexes (DDL + storage tuple encoding + prefix-matched seeks), edge IndexSeek for unbound-endpoint patterns (single-node, in-tx overlay, and cluster-routing scatter-gather), point / spatial indexes (Z-order cells + DDL + `withinbbox` and distance-radius planner rewrites for both node and relationship scope), and Neo4j 5 quantifier shorthand all shipped. `shortestPath(...)` and `allShortestPaths(...)` lower to a single layered-BFS operator that builds a multi-parent DAG and either short-circuits at the first complete path or enumerates every minimum-length path, locked in by paranoia tests covering cycles, parallel edges, undirected patterns, reverse direction, edge-type unions, and self-loops. APOC compatibility is now broad: scalar surfaces across nine namespaces, the `apoc.agg.*` aggregates, write + read procedures for `apoc.create.*` / `apoc.refactor.setType` / `apoc.meta.schema`, `apoc.periodic.iterate` on top of `CALL { } IN TRANSACTIONS` (sequential plus `parallel: true` / `concurrency: N` worker-pool dispatch), the full `apoc.path.*` traversal family (filter DSL + uniqueness modes + streaming), `apoc.cypher.run` / `doIt` for runtime query evaluation, `apoc.load.{json,csv}` + `apoc.export.{csv,json,cypher}.{all,query}` with a strict-default-off security surface, and the full `apoc.trigger.*` family — install / drop / list / start / stop with all four phases (before / after / afterAsync / rollback), the full APOC param surface (created/deleted/assigned/removed nodes / relationships / properties / labels), cluster-replicated registries through Raft + routing, and trigger-induced writes that recurse through the cluster commit path under a from-trigger suppression guard. Each procedure is feature-gated for slim builds; the umbrella `apoc` feature turns them all on.

## Style & Conventions
- Use `thiserror` for error types, `anyhow` in the binary crate.
- Prefer strong typing over stringly-typed interfaces.
- Each crate should have its own `Error` type. Public error enums carry `#[non_exhaustive]` so future variants don't count as breaking changes within a minor.
- Integration tests in each crate's `tests/` directory.
- Benchmarks with `criterion` for storage and query hot paths. The two suites live at `crates/meshdb-storage/benches/storage_bench.rs` (point lookup, single + batched inserts, adjacency scan, label / indexed-property seeks) and `crates/meshdb-rpc/benches/query_bench.rs` (end-to-end Cypher through `MeshService::execute_cypher_local`: `CREATE`, `MATCH` by label, indexed-property match, one-hop expansion, label-count aggregate). Run with `cargo bench -p meshdb-storage --bench storage_bench` or add `-- --quick` for a 30-second smoke run. Criterion writes HTML reports to `target/criterion/<bench_name>/report/index.html`. Benches are local-only — CI does not gate on them because noise would produce false alarms without a baseline + variance budget in place.
